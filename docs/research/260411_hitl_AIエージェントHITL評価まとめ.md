# AIエージェントのHITL評価まとめ

**調査日:** 2026-04-11

## ざっくり結論

AIエージェントの Human-in-the-Loop、つまり人間による確認・承認・判断を評価するときは、「何回発生したか」だけを見ても足りない。大事なのは、必要な確認を見逃していないか、逆に不要な確認で人間を疲れさせていないか、確認が起きるタイミングや並び方が妥当か、という全体像で見ること[LayerX エンジニアブログ](https://tech.layerx.co.jp/entry/2026/04/01/150000)。

この記事の主張を一言でいうと、HITL は「少ないほどよい」でも「多いほど安全」でもなく、リスクに応じて人間の介入を設計し、その設計が機能しているかを測るべき、という話。

## HITL とは何を見ているのか

HITL は、AIエージェントが勝手に進めると危ない場面で、人間が確認・承認・差し戻しをする仕組み。たとえば本番環境への変更、セキュリティ判断、法務・コンプライアンスが絡む判断、外部送信前の確認などが該当する。

評価対象としては、次のようなものがある。

| 観点 | 何を見るか |
| --- | --- |
| 回数 | 1回の作業で人間確認が何回起きたか |
| 必要性 | その確認は本当に必要だったか |
| 待ち時間 | 人間の返答待ちでどれくらい止まったか |
| 承認後の失敗 | 人間が承認したのに処理が失敗していないか |
| 見逃し | 本当は確認すべきだったのに確認されなかった場面はないか |

ただし、この記事のポイントは「これらを集計すれば終わり」ではない。集計値の先にある、リスクの重さや人間の判断品質まで見ようとしている。

## 重要ポイント1: 「見逃し」と「確認しすぎ」は同じ重さではない

HITL が少なすぎる場合は、本来人間が止めるべきリスクを見逃している可能性がある。これは障害、データ損失、セキュリティ事故、法的問題のように、被害が一気に大きくなる危険がある。

一方で、HITL が多すぎる場合は、人間の時間を使いすぎる、エージェントが待ち状態になる、というコストが中心になる。もちろん問題だが、多くの場合は「確認1件あたり何分」のように比較的見積もりやすい。

つまり、同じ「1件ズレた」でも意味が違う。

| ズレ方 | 何が起きるか | 評価の重み |
| --- | --- | --- |
| 確認が足りない | 危険な判断をAIが素通りする | 重く見る |
| 確認が多すぎる | 人間の工数と待ち時間が増える | 軽視はしないが、見逃しとは分けて見る |

そのため、単純に「理想回数との差」だけで評価すると危ない。安全に関わる確認では、不要な確認が少し残ることより、必要な確認を落とすことのほうを強く罰する設計にしたほうがよい。

## 重要ポイント2: 高リスクと低リスクを同じ指標で混ぜない

本番デプロイ承認の見逃しと、ドキュメント体裁確認の無駄打ちは、現場への影響がまったく違う。これをひとつの平均スコアに混ぜると、重要な差が薄まってしまう。

実務では、HITL をカテゴリに分けて評価するのがよさそう。

| カテゴリ例 | 評価方針 |
| --- | --- |
| セキュリティ、法務、本番操作 | 見逃しをかなり厳しく見る |
| データ更新、外部送信 | 見逃しを強めに見る |
| ドキュメント体裁、軽微な確認 | 無駄な確認も削減対象にする |

この考え方は、プロダクトの効率化だけでなく、コンプライアンスや安全性を「削ってはいけない制約」として扱うためにも重要。規制上必要な承認は、たとえ普段ほとんど差し戻されなくても、単純な効率化対象にはならない。

## 重要ポイント3: タイミングも見る

HITL は「起きたかどうか」だけでなく、「いつ起きたか」も重要。

早すぎる確認は、人間が判断する材料が揃っていない状態で確認を求めてしまう。結果として、差し戻しや追加確認が発生しやすい。

遅すぎる確認は、AIエージェントがかなり作業を進めたあとに止まるため、修正が必要になったときの手戻りが大きい。依存関係が増えた後だと、影響範囲も広がる。

| タイミング | 起きやすい問題 |
| --- | --- |
| 早すぎる | 判断材料不足、余計なやり取り |
| 遅すぎる | 大きな手戻り、依存先への波及 |
| ちょうどよい | 判断材料が揃い、戻りコストも抑えられる |

## 重要ポイント4: 個別のHITLだけ見ても足りない

個々のHITLが妥当に見えても、全体としては悪いパターンになっていることがある。

たとえば、1件ずつ見ると必要な確認でも、短時間に大量に来ると人間が疲れて後半は流れ作業のように承認してしまうかもしれない。逆に、単体では不要に見える確認でも、その確認があることで後続の重要判断の文脈が揃っていることもある。

この記事では、HITL の発生パターンを次の4つに分けている。

| パターン | 状態 | 見方 |
| --- | --- | --- |
| 集中型 | 特定のタイミングに確認が固まる | まとめて対応しやすいが、間の見逃しに注意 |
| 分散型 | 作業全体に均等に確認が出る | 継続的に関与できるが、割り込み疲れが起きやすい |
| フロントローディング型 | 初期に確認が多く、後半は減る | 最初に方向性を合わせて自律実行しやすい |
| ランダムバースト型 | 予測できないタイミングで急に増える | 人間が対応しにくく、設計見直しの優先度が高い |

成熟したエージェント運用では、ランダムバースト型を減らし、必要な初期確認を済ませたうえで後半は安全に自律実行できる状態を目指す、という考え方が使いやすい。

## 重要ポイント5: HITL同士はつながっている

HITL は独立したイベントではなく、前後の確認と影響し合う。

| 相互作用 | 例 | 評価のポイント |
| --- | --- | --- |
| 依存型カスケード | 最初の設計判断が後続の技術選定やテスト方針に影響する | 上流の確認ほど情報品質を高くする |
| 競合型干渉 | 短時間に大量の確認が来て注意力を奪い合う | 件数だけでなく疲労や承認品質を見る |
| 補完型シナジー | 同じ担当者が文脈を持つことで後続確認が速く正確になる | 担当者の一貫性にも価値がある |

特にエージェントが長いタスクを実行する場合、個別の承認ログだけでなく、確認の順序、間隔、担当者、後続結果まで追えるようにしておくと分析しやすい。

## 実務に落とすなら

まず、HITL をカテゴリ分けする。高リスクなものと低リスクなものを混ぜず、見逃しをどれくらい重く見るかをカテゴリごとに決める。

次に、HITL のログに少なくとも次の情報を残す。

| ログ項目 | 目的 |
| --- | --- |
| セッションID | どの実行中に起きたか追う |
| HITLカテゴリ | リスク別に評価する |
| 発生時刻 | 分布やバーストを見つける |
| 理由 | なぜ人間確認が必要だったか見る |
| 承認・差し戻し結果 | 判断品質を見る |
| 後続の成功・失敗 | 承認後の影響を見る |
| 担当者またはロール | 文脈共有や疲労を分析する |

可視化では、平均値だけを見ない。HITL 回数、発生時刻、応答時間、承認結果を時系列や散布図で見る。異常に固まっている箇所や、後半だけ承認が雑になっている箇所のほうが、平均値より改善の手がかりになる。

## このプロジェクトでの示唆

MAF を使ったエージェント実行管理基盤で HITL を扱うなら、単に「HITLイベントを保存する」だけではなく、あとから評価できる形で保存したい。

特に、HITL のカテゴリ、発生理由、発生タイミング、応答時間、承認後の結果、セッション内での順序は重要。これらがあれば、エージェントの自律性を上げるべき箇所と、人間確認を残すべき箇所を分けて判断しやすくなる。

設計上は、HITL を減らすことを直接のKPIにしないほうがよい。高リスク領域では見逃しを減らすこと、低リスク領域では不要な確認を減らすこと、全体ではランダムな確認バーストを減らすことを別々に追うのがよさそう。

## ひとことで覚える

HITL 評価は「人間確認を何回減らせたか」ではなく、「危ない見逃しを防ぎつつ、人間の判断力を無駄遣いしていないか」を見るためのもの。
